# Version Baseline

Verification date: 2026-03-26  
Verification rule: official documentation, official release notes, and official project pages are the primary source of truth.  
Selection rule: stable releases only; no alpha, beta, canary, RC, preview, experimental, or nightly versions for the baseline.

## Version Policy

- Compatibility beats novelty when SDK-managed ecosystems are involved.
- Where Expo or Supabase own compatibility, the project follows the supported version line instead of forcing newer upstream packages.
- Exact pins are preferred for root tooling and direct dependencies.
- SDK-managed mobile dependencies may use the exact range emitted by `expo install` for the chosen stable SDK line.
- If a version changes intentionally later, update this file and `docs/DECISIONS.md` in the same session.

## Notes On Deferred Exact Pins

Some ecosystem owners publish compatibility lines more clearly than exact patch guidance. For those cases, the baseline intentionally locks to a supported stable line now and requires an explicit exact pin when the dependency is first introduced in code.

This still applies to:

- `@supabase/supabase-js`
- Zustand
- Zod
- provider-managed PostgreSQL and PostGIS runtime components

## Phase 1 And Phase 2 Version Decisions Summary

- Mobile baseline remains Expo SDK 55 and is now reflected in a real `apps/mobile` package.
- Expo-managed mobile companion packages were installed through `expo install` and recorded above.
- TypeScript is pinned to `5.9.3` because the official TypeScript site still treats 6.0 as RC-only.
- The root tooling stack is now pinned to ESLint `10.1.0`, `typescript-eslint` `8.57.2`, Prettier `3.8.1`, and Vitest `4.0.16`.
- The future admin-web baseline remains `16.2.1`, matching the current official Next.js docs "Latest Version" value at correction time.
- The contributor runtime baseline was corrected from Node `22.22.2` to the current official Latest LTS, Node `24.14.1`, across `.nvmrc`, `.node-version`, `package.json`, and CI.
- Phase 2 added `expo-font@~14.0.9` for bundled local font loading and locked the active font baseline to Heebo only.
- Phase 2 added `@testing-library/react-native@13.3.3`, `react-test-renderer@19.2.0`, and `@types/react@19.2.0` for the first shared UI primitive tests.
- The installed React Native testing-library package remains the intended long-term mobile component test library, but the executable Phase 2 baseline currently routes through a small local compatibility harness to stay within the node-only Vitest constraint.
